Introducción a Git

Sistema de control de versiones (VCS)

¿Qué es?

Un sistema de control de versiones registra los cambios realizados en uno o varios archivos a lo largo del tiempo. Gracias a esto, es posible consultar el historial de un proyecto, recuperar versiones anteriores y trabajar de manera más ordenada.

¿Para qué sirve?

  • Volver a versiones anteriores de uno o varios archivos.
  • Recuperar una versión previa del proyecto completo.
  • Comparar cambios realizados en distintos momentos.
  • Identificar quién modificó por última vez una parte del código.
  • Detectar quién y cuándo introdujo un error.
  • Restaurar archivos dañados o eliminados.

Tipos de VCS

Los sistemas de control de versiones pueden clasificarse en tres grandes tipos:

  • Local
  • Centralizado
  • Distribuido

VCS local

En un sistema de control de versiones local, el historial de cambios se almacena en la misma computadora del usuario. Es una solución simple, pero presenta limitaciones para el trabajo colaborativo y mayores riesgos ante fallos del equipo o pérdida de información.

VCS centralizado

En un sistema centralizado, existe un servidor único que almacena el historial del proyecto y desde el cual los usuarios obtienen y envían cambios. Facilita la administración del trabajo en equipo, aunque depende fuertemente de la disponibilidad del servidor central.

VCS distribuido

En un sistema distribuido, cada usuario posee una copia completa del repositorio, incluido su historial. Esto mejora la redundancia, permite realizar muchas operaciones de forma local y favorece el trabajo colaborativo sin depender permanentemente de un servidor central.

Git

¿Qué es?

Git es un sistema de control de versiones distribuido. Fue diseñado para trabajar de manera rápida, segura y eficiente, incluso en proyectos grandes y con múltiples colaboradores.

Objetivos de diseño

  • Alto rendimiento.
  • Diseño simple.
  • Soporte para desarrollo no lineal.
  • Funcionamiento completamente distribuido.
  • Capacidad para manejar proyectos de gran tamaño.

Fundamentos de Git

Las ideas principales sobre las que se apoya Git son las siguientes:

  • Trabaja con copias instantáneas, no solo con diferencias.
  • La mayoría de las operaciones son locales.
  • Garantiza la integridad de los datos.
  • Los archivos atraviesan distintos estados durante el trabajo.

Almacenamiento de datos

La primera imagen representa un modelo de almacenamiento basado en copias diferenciadas: cada nueva versión guarda únicamente los cambios respecto de la versión anterior.

La segunda imagen muestra el enfoque basado en copias instantáneas, que es el utilizado por Git: en cada versión se registra una instantánea del estado de los archivos del proyecto. Esta decisión permite manejar el historial de manera más eficiente y confiable.

Fundamentos de Git (continuación)

Copias instantáneas, no diferencias

Git guarda el estado de los archivos en distintos momentos del tiempo. En lugar de almacenar únicamente las diferencias entre versiones, registra instantáneas del proyecto. Esto permite trabajar con el historial de forma más eficiente y confiable.

Operaciones locales

En Git, muchas operaciones se realizan sobre la copia local del repositorio. Esto hace que acciones como consultar el historial, comparar cambios o crear commits sean rápidas y no dependan de una conexión permanente a Internet.

Integridad: suma de verificación (SHA-1)

Git identifica cada objeto mediante una suma de verificación calculada con el algoritmo SHA-1.

  • Se representa como una cadena de 40 caracteres hexadecimales (0-9 y a-f).
  • Ejemplo: 24b9da6552252987aa493b52f8696cd6d3b00373

Estados

En Git, un archivo puede encontrarse en alguno de estos estados:

  • Confirmado (committed): los datos ya están almacenados de manera segura en la base de datos local del repositorio.
  • Modificado (modified): el archivo fue editado, pero sus cambios todavía no se incorporaron al próximo commit.
  • Preparado (staged): el archivo modificado fue marcado para incluirse en la próxima confirmación.

Estados: flujo de trabajo

La imagen representa los tres espacios principales del flujo de trabajo de Git. El working directory es el directorio de trabajo donde se crean y modifican los archivos. La staging area es el área de preparación donde se seleccionan los cambios que formarán parte del próximo commit. El .git directory (repository) es la base de datos local del repositorio, donde Git almacena de manera permanente la información confirmada.

Comandos básicos

Configuración inicial

Definir el nombre del autor:

git config --global user.name "Nombre y apellido"

Definir el correo electrónico del autor:

git config --global user.email "correo@email.com"

Definir el editor de texto predeterminado:

git config --global core.editor notepad

Mostrar la configuración actual:

git config --list

Crear un repositorio

Inicializar un nuevo repositorio Git:

git init

Consultar el estado

Mostrar el estado completo del repositorio:

git status

Mostrar una versión resumida del estado:

git status -s

Estados de archivos

La imagen muestra los posibles estados por los que puede pasar un archivo dentro del repositorio. Un archivo untracked es un archivo nuevo que Git todavía no está siguiendo. Un archivo unmodified no tiene cambios respecto de la última versión confirmada. Un archivo modified fue editado, pero esos cambios aún no fueron preparados. Un archivo staged ya fue agregado al área de preparación y está listo para formar parte del próximo commit.

Otros comandos frecuentes

Agregar archivos al área de preparación

git add archivo

Confirmar cambios

git commit -m "mensaje"

Agregar y confirmar en un solo paso

git commit -a -m "mensaje"

Renombrar archivos

git mv nombre_viejo nombre_nuevo

Modificar el último commit

Se utiliza, por ejemplo, para corregir el mensaje del último commit o agregar archivos que faltaron incluir:

git commit --amend

Quitar un archivo del área de preparación

git restore --staged archivo

Descartar cambios en un archivo modificado

git restore archivo

Consultar el historial

Mostrar el historial completo:

git log

Mostrar solo el último commit:

git log -1

Mostrar los últimos dos commits con sus diferencias:

git log -p -2

Mostrar el historial de un archivo específico:

git log archivo

Clonar un repositorio

git clone repositorio

Trabajar con repositorios remotos

Ver los repositorios remotos configurados:

git remote -v

Agregar un repositorio remoto:

git remote add nombre repositorio

Descargar cambios desde un repositorio remoto:

git fetch nombre

Descargar y fusionar cambios del repositorio remoto:

git pull origin master

Enviar cambios al repositorio remoto:

git push origin master
