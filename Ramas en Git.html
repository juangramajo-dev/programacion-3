Ramas en Git

Introducción

En el apunte de introducción vimos que Git:

  • Es un sistema de control de versiones distribuido
  • Guarda instantáneas del proyecto
  • Identifica cada objeto mediante una suma de verificación
  • Registra cambios por medio de commits

Sobre esa base, las ramas permiten trabajar en distintas líneas de desarrollo sin mezclar cambios de manera inmediata. Esto es especialmente útil cuando se quiere probar una idea, desarrollar una funcionalidad nueva o corregir un error sin afectar la versión principal del proyecto.

En muchos sistemas de control de versiones crear una rama puede implicar copiar gran parte del proyecto. En Git, en cambio, las ramas son livianas y rápidas de crear. Por eso forman parte del flujo de trabajo habitual.

¿Que es una rama?

Una rama es un puntero que señala un commit. A medida que se crean nuevos commits, ese puntero avanza.

Dicho de otro modo: una rama no es una copia completa del proyecto, sino una referencia a una secuencia de commits.

Por defecto, Git suele crear una rama principal llamada master o main, según la configuración del repositorio. En los ejemplos de este apunte se mantiene el nombre master porque así aparece en el material de referencia.

¿Por qué Git puede manejar ramas tan rápido?

Recordando el apunte de introducción, Git no guarda solo diferencias entre versiones. Guarda instantáneas del estado del proyecto y relaciones entre commits.

Cada commit almacena:

  • Una instantánea de los archivos preparados
  • Metadatos, como autor y mensaje
  • Una referencia a su commit padre
  • En caso de fusión, referencias a varios commits padre

Eso permite que una rama sea simplemente un puntero móvil hacia un commit. Crear una rama, entonces, es una operación muy barata.

Ejemplo conceptual

Supongamos que un proyecto tiene tres archivos. Se agregan al área de preparación y luego se confirma el estado actual:

git add README test.rb LICENSE
git commit -m 'Commit inicial del proyecto'

Luego de ese commit, Git registra los objetos necesarios para reconstruir el estado del proyecto y la rama principal pasa a apuntar a ese commit.

Si se realizan nuevos commits, Git enlaza el commit nuevo con el anterior.

HEAD: la referencia de la rama actual

Git usa una referencia especial llamada HEAD para indicar en que rama estamos trabajando en este momento.

Si estamos en master, entonces HEAD apunta a master. Si cambiamos a otra rama, HEAD pasa a apuntar a esa nueva rama.

Crear una rama nueva

Para crear una rama nueva se utiliza:

git branch testing

Este comando crea la rama testing, pero no cambia automáticamente a esa rama. La nueva rama apunta al mismo commit en el que estábamos trabajando.

Podemos ver las referencias con:

git log --oneline --decorate

Ejemplo de salida:

f30ab (HEAD, master, testing) add feature #32 - ability to add new
34ac2 fixed bug #1328 - stack overflow under certain conditions
98ca9 initial commit of my project

En este caso, master y testing apuntan al mismo commit.

Cambiar de rama

Para pasar a otra rama se utiliza:

git checkout testing

Ahora HEAD apunta a testing.

Si hacemos cambios y confirmamos:

git commit -a -m 'made a change'

la rama testing avanza, mientras que master permanece donde estaba.

Volver a la rama principal

Para regresar a master:

git checkout master

Cuando se cambia de rama, Git actualiza los archivos del directorio de trabajo para que coincidan con la última instantánea confirmada en esa rama.

Esto significa que cambiar de rama no solo modifica un puntero interno: también puede cambiar el contenido visible de los archivos en el proyecto.

Si Git detecta cambios que impedirían hacer el cambio de rama de manera segura, no lo permite hasta resolver la situación.

Historial divergente

Si después de volver a master hacemos otro commit:

git commit -a -m 'made other changes'

entonces el historial diverge: master sigue un camino y testing sigue otro.

Esto permite desarrollar lineas de trabajo independientes sin que una interfiera con la otra.

Ver el historial de ramas

Para observar mejor la divergencia entre ramas:

git log --oneline --decorate --graph --all

Ejemplo de salida:

* c2b9e (HEAD, master) made other changes
| * 87ab2 (testing) made a change
|/
* f30ab add feature #32 - ability to add new formats to the
* 34ac2 fixed bug #1328 - stack overflow under certain conditions
* 98ca9 initial commit of my project

La opción --graph dibuja las bifurcaciones del historial y --all muestra todas las ramas.

Ideas clave

  • Una rama es un puntero a un commit.
  • HEAD indica la rama actual.
  • git branch crea ramas.
  • git checkout permite cambiar de rama.
  • Cada rama puede avanzar con sus propios commits.
  • Cambiar de rama puede modificar los archivos del directorio de trabajo.

Procedimientos Básicos para Ramificar y Fusionar

En esta parte se muestra un flujo típico: trabajamos una funcionalidad en una rama, aparece una urgencia, la resolvemos en otra rama y luego retomamos el trabajo original.

Escenario de trabajo

  1. Se crea una rama para el issue #53.
  2. Se avanza con commits en esa rama.
  3. Surge un error crítico y se crea una rama hotfix.
  4. Se fusiona hotfix en master.
  5. Se retoma iss53 y luego se fusiona en master.

Ramificación para una funcionalidad

Crear y cambiar a la rama iss53:

git checkout -b iss53

Atajo equivalente:

git branch iss53
git checkout iss53

Trabajar y confirmar cambios:

vim index.html
git commit -a -m 'Agregar pie de página [issue 53]'

Interrupción por un hotfix

Antes de cambiar de rama conviene tener el directorio limpio (sin cambios sin confirmar) para evitar conflictos al hacer checkout.

Volver a master:

git checkout master

Crear rama de urgencia, corregir y confirmar:

git checkout -b hotfix
code index.html
git commit -a -m 'Corregir correo electrónico'

Fusionar hotfix en master:

git checkout master
git merge hotfix

Si aparece Fast-forward, Git solo avanza el puntero de la rama: no necesita crear un commit de merge porque no hubo historia divergente.

Eliminar la rama que ya no se usa:

git branch -d hotfix

Retomar el trabajo pendiente:

git checkout iss53
vim index.html
git commit -a -m 'Terminar el nuevo pie de página [issue 53]'

Fusión de una rama de funcionalidad

Cuando iss53 está lista, se fusiona en master:

git checkout master
git merge iss53

En este caso suele haber fusión de tres vías (three-way merge): Git busca el ancestro común y crea un commit de merge con más de un padre.

Luego se puede borrar la rama:

git branch -d iss53

Conflictos de fusión (merge conflicts)

Si ambas ramas modifican la misma parte de un archivo, Git no puede decidir automáticamente:

git merge iss53

Salida típica:

Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

Ver archivos en conflicto:

git status

Salida típica:

On branch master
You have unmerged paths.
  (fix conflicts and run "git commit")
 
Unmerged paths:
  (use "git add <file>..." to mark resolution)
 
    both modified:   index.html

Git marca el archivo con separadores de conflicto:

<<<<<<< HEAD:index.html
# <div id="footer">contact : email.support@github.com</div>
=======
<div id="footer">
 please contact us at support@github.com
</div>
>>>>>>> iss53:index.html

Después de editar y dejar una versión final (sin <<<<<<<, =======, >>>>>>>), se marca como resuelto:

git add <file>...

También se puede usar herramienta gráfica:

git mergetool

Ayuda y configuración de mergetool:

git mergetool --tool-help
git help config

Si todo quedó resuelto, se finaliza la fusión:

git status
git commit
